# Changelog

## 0.4.0

**Methodology improvements — Stage 4 split into 4a (Test Optimization) + 4b (API Compliance).**

### New

- `skills/element-interactions/references/test-optimization.md` — single source of truth for Stage 4a's 6-check protocol, 2-of-2 API-shortcut gate, helper templates (`resetState`, `setAuthCookie`, `seedCart`, `dismissBanners`), and structured-return format.
- `skills/journey-mapping/references/test-infrastructure-probe.md` — protocol for capturing auth model, reset endpoints, persistent banners, mutation endpoints, and stable seed resources during Phase 1 crawl.
- New `## Test Infrastructure` section in `tests/e2e/docs/app-context.md` (auto-generated by `journey-mapping`).
- New `UI-covers:` field on each journey block in `journey-map.md` — registry that gates API-shortcut decisions.
- Whole-suite re-run gate at orchestrator exits (`test-composer` Step 7, `coverage-expansion` per-pass).

### Changed

- `skills/element-interactions/SKILL.md`: Stage 4 splits into Stage 4a (Test Optimization) + Stage 4b (API Compliance — content unchanged). Orchestrator points to references rather than inlining the protocol.
- `skills/test-composer/SKILL.md`: Step 6 splits into Step 6a (Test Optimization) + Step 6b (API Compliance). Canonical return schema gains `stage_4a` and `whole_suite_gate` blocks.
- `skills/journey-mapping/SKILL.md`: Phase 1 invokes the Test Infrastructure probe; Phase 4 documents the new `UI-covers:` field with canonical flow-name vocabulary.
- `skills/onboarding/SKILL.md`: Phase 1 scaffold writes `tests/fixtures/base.ts` with four `HELPER SLOT` markers Stage 4a populates on demand.
- `skills/coverage-expansion/SKILL.md`: per-pass exit runs the whole-suite re-run gate before advancing.

### Motivation

The 2026-04-29 onboarding A/B test (`onboarding-ab-test/`, comparing element-interactions `0.2.9` vs locally-packed HEAD on `book-hive`) surfaced that the prior methodology shipped a suite with **79/109 passing** (10 timeouts + 20 cascade-skips) due to integration-time pollution: 15 spec files hardcoded `book-001` and the suite had no `beforeEach` reset, so book stock depleted across the run. Per-pass `stabilize` confirmed each batch but the cumulative suite was never re-run end-to-end. Stage 4a's six checks plus the whole-suite re-run gate close that blind spot.

The OLD arm in the same A/B used `test.beforeEach(POST /api/reset)` and shipped 46/47 passing — that's the discipline this methodology change re-establishes and generalizes.
